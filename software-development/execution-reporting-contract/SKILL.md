# 执行-汇报契约

> 原则：用户永远不应该在任务执行期间陷入信息真空。任何工具调用后，如果还有下一步，必须先汇报进度；如果是最后一步，必须立即汇报结果。

## 三条硬性规则

### 规则 1：执行前必须声明
任何预计耗时 > 30 秒的操作，在第一个 tool call 之前，必须向用户发送一条消息：
- 要做什么
- 预计多久
- 分几个阶段

### 规则 2：每个阶段结束必须汇报
将长任务拆分为可独立验证的阶段。每个阶段结束后，即使后面还有阶段，也必须先发送汇报：
- 阶段名称
- 成功/失败
- 关键产出（路径、文件名、git commit 等）

### 规则 3：最终结果必须在同一个 turn 内发出
如果任务在当前 turn 中完成，不能以"等我检查一下"或类似理由延迟回复。必须在收到最后一个 tool result 后，立即生成最终汇报并发出。

## 违规后果
- 用户在任务执行期间收不到消息
- 用户不知道任务是否完成
- 用户不知道失败了哪一步
- session 中断导致永久性信息丢失

## 应对策略（当前可行）
1. 所有涉及 > 2 个 tool calls 的任务，必须在每个完整的轮次交付进度
2. 如果任务分为多个阶段，每个阶段用独立的自然语言消息分割，不能压缩成一条
3. 即使最后一个阶段失败，也必须汇报失败原因和已完成的部分
